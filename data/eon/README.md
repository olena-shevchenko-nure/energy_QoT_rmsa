# EON Data Generation

This directory contains a reproducible data-generation pipeline for experiments
on energy-aware dynamic resource allocation in elastic optical networks. The v1
pipeline targets a compact NSFNET dataset for GNN + CNN + DQN experiments with
deterministic feasible-action filtering, Top-N padding, and `candidate_mask`.

## Why NSFNET First

The default topology is a DeepRMSA-compatible NSFNET 14-node / 22-link /
44-directed-link topology with 100 slots per directed fiber. It is small enough
for fast smoke tests, yet rich enough to exercise routing, modulation reach,
spectrum fragmentation, candidate masking, supervised ranking, and offline RL
warm-start data.

## Regeneration

From the repository root, install the lightweight data dependencies, then run:

```bash
python scripts/data/generate_eon_data.py --config configs/data/nsfnet_smoke.yaml --output data/eon/generated/nsfnet_smoke
python scripts/data/validate_eon_data.py --dataset data/eon/generated/nsfnet_smoke
python scripts/data/summarize_eon_data.py --dataset data/eon/generated/nsfnet_smoke
```

The module entrypoint is also available:

```bash
python -m cse2026.data_generation.generate --config configs/data/nsfnet_smoke.yaml --output data/eon/generated/nsfnet_smoke
```

The larger MVP dataset can be regenerated locally with:

```bash
python scripts/data/generate_eon_data.py --config configs/data/nsfnet_mvp.yaml --output data/eon/generated/nsfnet_mvp
```

The full experiment template is intentionally not generated by default:

```bash
python scripts/data/generate_eon_data.py --config configs/data/nsfnet_full_template.yaml --output data/eon/generated/nsfnet_full
```

## Files And Schemas

Generated datasets contain:

- `traffic/*.parquet`: one request row per state/request.
- `candidates/*.parquet`: Top-N candidate rows with dummy padding and `candidate_mask`.
- `candidates/*_full.parquet`: all feasible generated candidates before Top-N truncation.
- `gnn/*_graphs.npz` and `gnn/*_routes.parquet`: graph tensors and route/modulation labels.
- `cnn/*_tensors.npz` and `cnn/*_index.parquet`: route-slot candidate tensors and labels.
- `dqn/*_transitions.parquet`: replay-buffer-like transitions collected by a non-oracle policy.
- `manifest.json`, `checksums.sha256`, and reports under `reports/`.

Schema JSON files are emitted under `gnn/`, `cnn/`, and `dqn/`.

## Candidate Generation And Safety

For each request, the generator computes K shortest paths and checks every
route/modulation pair. It builds a route-level slot availability mask, identifies
contiguous free blocks, and emits first-fit, best/exact-fit, last-fit, and a
small number of deterministic random starts. Candidates are feasible only when
spectrum continuity, contiguity, no-overlap, modulation reach, QoT margin, and
delay-bound checks all pass.

Feasible candidates are sorted by a deterministic fragmentation, gap, energy,
delay, QoT, and width score. The top `N_max` real candidates are retained and
padded with dummy rows. `candidate_mask = 1` marks real feasible candidates and
`candidate_mask = 0` marks padding. Losses and RL action selection must ignore
padding candidates.

## Experiment Use

Classical baselines can read `traffic`, `topology`, and the full candidate files
to replay SP-FF, KSP-FF, KSP-BM-FF, Energy-aware KSP-BM-FF, Exact-Fit, and
First-Last-Fit policies.

Supervised candidate rankers can use `candidates/*_full.parquet` or the padded
Top-N `candidates/*.parquet` files with `J_total`, feasibility labels, and route,
slot, energy, QoT, delay, and fragmentation features.

GNN pretraining uses `gnn/*_graphs.npz` for graph, request, and global state
features, plus `gnn/*_routes.parquet` for route/modulation labels and scores.

CNN route-slot pretraining uses `cnn/*_tensors.npz` and `cnn/*_index.parquet` to
predict placement quality, fragmentation deltas, small-gap creation, and the
heuristic placement score.

DQN and Double DQN warm-start experiments can read `dqn/*_transitions.parquet`
with selected action indices, rewards, candidate-mask validity flags, block
labels, and Q-head pretraining labels. The collection policy is intentionally
non-oracle: mostly best-fit/low-fragmentation with energy tie-breaks, plus
softmax and random exploration.

Top-N sensitivity experiments can regenerate the same traffic with different
`n_max` values or filter the full candidate tables at analysis time.

## Known Limitations

- QoT is a reach-based proxy in v1.
- The energy model is approximate and uses amplifier, transponder, and ROADM
  switch costs.
- Larger datasets should be regenerated locally or stored with Git LFS/DVC
  rather than committed as large binary artifacts.
